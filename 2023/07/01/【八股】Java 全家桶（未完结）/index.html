类加载

1. 类加载过程？

加载过程：

  1. 加载：
    • 根据全限定名，获取二进制**字节流**；
    • 将字节流里的静态存储结构转为方法区的**运行时数据结构**；
    • 堆中生成该类的 Class 对象，作为方法区访问堆中数据的**入口**。
  2. 连接：
    1. 验证：验证 .class 文件符合**虚拟机规范**，内容包括文件头、语法、语义、符号引用；
    2. 准备：为方法区中类的静态变量分配内存，初始化默认值（**零值**），特别地，如果变量被 static final 修饰，则会初始化为初值，放入常量池；
    3. 解析：**符号引用转为直接引用**。
  3. 初始化：静态变量初始化为**初值**。

后续：

  1. 使用：类访问方法区中数据结构的接口，对象则是堆中的数据；
  2. 卸载：Class 对象被垃圾回收，自定义类加载器被回收。

【知识补充】：

  1. 卸载类的条件：
    1. 堆中不存在该类的实例对象；
    2. 该类没有被其他任何地方引用；
    3. 该类的类加载器的实例已被垃圾回收。

【相关追问】：

  1. 符号引用？直接引用？
    • 符号引用：一组符号来描述目标，可以是任何字面量；
    • 直接引用：直接指向目标的指针、相对偏移量或间接定位到目标的句柄。
  2. 虚拟机生命周期的结束方式？
    • 执行 System.exit()；
    • 正常结束；
    • 程序异常或错误；
    • 操作系统出现错误。
  3. 加载顺序是否固定？
    加载、验证、准备、初始化的顺序确定，但解析的顺序不确定，因为 Java 支持**动态绑定，所以解析可能发生在初始化之后。另外，这个顺序是开始顺序，而非结束的顺序，一般是交叉进行**的。
  4. 验证阶段必须存在吗？
    不是。如果引用的类需要被反复验证，那么可以用 -Xverifynone 参数关闭大部分的类验证，缩短验证时间。

2. 动态绑定的好处？

比如 Person p = new Student(); // Student 继承自 Person，Person 中有方法 eat() ：

  • 多态：p 指向 Student 的引用，调用的方法是 Student 的方法；
  • 扩展性高：对 Person 新增的多个子类都可以使用 eat() 方法；
  • 解耦：eat() 方法不依赖子类；
  • 灵活：可以根据运行时的具体条件，**灵活选择调用的方法**。

3. 加载 .class 文件的方式？

  • 本地直接加载；
  • 网络下载 .class 文件；
  • 从 zip、jar 等归档文档中加载；
  • 从专有数据库中提取；
  • .java 文件动态编译为 .class 文件。

4. 类加载器有哪些？层次如何？

从上至下有四种，之间通过组合的方式表示父子关系：

  1. 启动类加载器：由 C++ 实现，没有父级，主要加载 **JDK 核心类库**和指定的路径下的类；
  2. 扩展类加载器：主要加载 lib/ext 路径下的 **jar 包**和指定的路径下的类；
  3. 应用程序类加载器：面向用户的类加载器，主要加载当前 classpath 下的所有 **jar 包和类**，没有自定义类加载时默认的类加载器；
  4. 自定义类加载器：继承 ClassLoader 抽象类，满足特殊需求，比如对 .class 文件加密后，用自定义的类加载器解密。

【知识补充】：

  1. 类加载器作用：**加载 .class 文件到 JVM 中**，即在内存中生成一个代表该类的 Class 对象；
  2. 启动类加载器是 JVM 的一部分，其他的都是 JVM 外部实现的，且**全都继承自 ClassLoader 抽象类**；
  3. 每个 ClassLoader 可以通过 getParent() 来获取父 ClassLoader，如果为 null 则说明该类是由启动类加载器加载的（结果为 null 是因为它是 C++ 实现的）；
  4. JVM 如何判定两个 Java 类相同：
    全限定名一样，并且类加载器一样。

5. 类加载器加载规则？

不会一次性加载所有的类，而是**动态加载，类在加载时会先判断该类是否已被加载，如果已被加载，则直接返回，即，相同二进制名称的类只会被加载一次**。另外，已加载的类被放在 ClassLoader 抽象类中。

6. ❓类加载的具体方法？区别？

  1. 命令行启动应用时候由 JVM 初始化加载；
  2. 由 Class.forName() 方法动态加载 .class 文件到 JVM，默认执行类中的静态块，也可自定义不执行；
  3. 由 ClassLoader.loadClass() 方法动态加载 .class 文件到 JVM，默认不执行类中的静态块，在 new 实例的时候才会执行。

7. 类加载机制有哪些？

  • 全盘委托：一个类加载器在加载某个类时，这个**类所依赖和引用的其他的类**也都有该类加载器加载；
  • 父类委托：优先让父加载器尝试加载，无法加载时再由自己去加载；
  • 缓存机制：加载过的类**放入缓存**，优先从缓存中寻找目标类；
  • 双亲委派：类加载器将收到的加载请求委托给父加载器，**逐层向上**直到启动类加载器，父加载器无法加载时再由自己去加载。

8. 双亲委派机制的具体过程？

  1. 应用程序类加载器将加载请求委派给扩展类加载器；
  2. 扩展类加载器继续将加载请求委托给启动类加载器；
  3. 启动类加载器处理请求，如果无法加载，则交由扩展类加载器加载；
  4. 扩展类加载器处理请求，如果无法加载，则交由应用程序类加载器；
  5. 最后再由应用程序类加载器处理类加载请求，如果仍无法加载，则抛出 ClassNotFoundException 异常。

9. 为什么需要双亲委派机制？

这个问题也可改为双亲委派机制的优势：

  • **避免重复**加载：加载过的类不会被重复加载，防止内存中出现多份同样的字节码，节省资源；
  • **安全**：可以保证核心 API 不会被篡改，也就是说我自己改写的核心类不会被执行；
  • 类库**隔离**：Java 程序可能有多个模块，每个模块都有自己的类库，该机制可以避免不同模块之间的类加载发生冲突；
  • 结构性**层次**：通过组合的方式建立父子关系，有助于管理类加载的顺序和依赖关系；
  • **避免版本冲突**：如果同一个类在不同类加载器中有不同版本，该机制可以优先选择加载最早的版本，避免了版本冲突。

10. 双亲委派机制如何实现的？

loadClass() 源码抽象出来流程如下：

  1. 检查该类**是否已被加载**过：findLoadedClass(name) —> 返回值是否为 NULL；
  2. 如果未被加载过：则调用**父类加载的 loadClass()** 方法进行加载；
  3. 如果父类加载器为 NULL：检查这个类是否由**启动类加载器**加载；
  4. 如果父类加载和启动类加载器都无法完成类加载：**调用自己的 findClass()** 方法完成类加载。

【知识补充】：

  1. 哪个方法实现了双亲委派机制？
    loadClass()。想要破坏双亲委派机制，重写该方法即可。
  2. loadClass()、findClass()、defineClass() 的区别？
    • loadClass()：负责类加载，实现了双亲委派机制；
    • findClass()：根据名称或位置加载 .class 字节码文件；
    • defineClass()：将字节码转为 Class。

11. 如何实现自定义类加载器？

  • **不打破**双亲委派机制：继承 ClassLoader 抽象类，重写 findClass() 方法；
  • **想打破**双亲委派机制：继承 ClassLoader 抽象类，重写 loadClass() 方法。

12. 双亲委派机制被打破的例子？

  • Tomcat 等 Web 容器：一个 Web 容器里面可能有多个应用程序，不同的应用程序可能依赖同一个类库的不同版本，而同一个类库的不同版本的路径名可能是一样的。如果使用双亲委派机制，就无法加载多个版本的类库。取而代之的是提供了隔离机制，为每个容器单独提供一个类加载器；
  • 一些模块化技术；
  • 实现热插拔部署工具。

运行时数据区（JVM 内存结构）

1. 介绍一下运行时数据区？

  • 线程私有：程序计数器、虚拟机栈、本地方法栈；
  • 线程共享：堆、方法区（JDK1.8 之后被移到本地内存成为元空间）。

2. 程序计数器？

作用：

  • 字节码通过改变程序计数器的值来选取下一条要读取的指令，实现代码的**流程控制**，比如：循环、分支、跳转、异常处理、线程恢复等；
  • **多线程**环境下，程序计数器记录当前线程执行的位置，实现线程之间的切换或恢复。

【知识补充】：

  1. 它是运行速度最快的区域，占用空间很小，唯一不会发生内存溢出的地方；
  2. 生命周期和线程的生命周期一致。

【相关追问】：

  1. 为什么要让程序计数器私有？
    多线程环境下，CPU 不停地做任务切换，执行的方法需要经常中断或恢复，所以每个线程都要有属于自己的程序计数器。

3. 虚拟机栈？

每调用一个方法，就有一个对应的栈帧被压入栈；每一个方法调用结束后，都会有一个栈帧被弹出。

虚拟机栈的构成：

  • 局部变量表：存储方法参数与方法体内的局部变量（包括各种数据类型和对象引用），存储单元是“槽”；
  • 操作数栈：存储方法执行过程中产生的中间结果和计算过程中的临时变量；
