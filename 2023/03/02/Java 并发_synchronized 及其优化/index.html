Java 并发_synchronized 及其优化

发表于2023-03-02|更新于2023-03-02|JavaSE

内容包括：

  1. 概览
  2. 原理
  3. 优化
  4. 与 Lock 对比

如何使用 synchronized

注意点：

  • 一把锁只能同时被一个线程获取，没有获取锁的线程只能等待
  • 每个实例都有一把锁（this），但锁对象如果是 *.class（也就是 Class 对象） 或 修饰的是 static 方法，那么所有对象共用同一把锁
  • 修饰方法时，无论如何退出，都会释放锁
  • 是非公平锁

对象锁

  • 修饰方法：方法锁（默认锁的对象是 this）
  • 修饰同步代码块：同步代码块锁（可以自己指定锁对象）

类锁

  • 修饰静态方法
  • 修饰指定的 Class 对象

原理分析

monitorenter 和 monitorexit 指令，会让对象锁计数器+1 或-1。

原理：

  • 每个对象在同一时间只能与一个 monitor(锁) 相关联
  • 每个 monitor 在同一时间只能被同一线程获得
  • 当对象尝试获得 monitor(锁) 时，monitorenter 有三种情况：
    • monitor 计数器为 0，此时线程获得锁，计数器+1，其他线程等待
    • monitor 如果已经拥有锁的所有权，又重入了这把锁，计数器再继续+1
    • 如果锁已被其他线程获取，则等待释放

monitorexit 指令：释放对 monitor 的所有权，计数器-1，直到为 0 时，才会释放锁。

  • 任意线程对 Object 的访问，都需要获得 monitor

  • 如果获得失败，则进入同步状态，此时线程处于阻塞状态

  • 锁释放后，同步队列中的线程则有机会重新获取该 monitor

  • 可重入原理：加锁次数计数器

  • 保证可见性原理：内存模型和 happens-before 原则

synchronized 锁优化

  • 锁粗化(Lock Coarsening)：也就是减少不必要的紧连在一起的 unlock，lock 操作，将多个连续的锁扩展成一个范围更大的锁
  • 锁消除(Lock Elimination)：通过运行时 JIT 编译器的逃逸分析来消除一些没有在当前同步块以外被其他线程共享的数据的锁保护，通过逃逸分析也可以在线程本的 Stack 上进行对象空间的分配（同时还可以减少 Heap 上的垃圾收集开销）

锁的类型

JDK1.6
锁膨胀方向： 无锁 → 偏向锁 → 轻量级锁 → 重量级锁 (此过程是不可逆的)

自旋锁与自适应自旋锁

自旋锁在前一篇Java 中所有的锁 中已经介绍了，这里讲一下自适应自旋锁。

JDK1.6 引入自适应自旋锁，自旋次数不再固定，而是由前一次在同一个锁上的自旋时间以及锁的拥有者的状态来决定的：

  • 如果同一个锁对象上，自旋等待刚刚成功，并且持有锁的线程正在运行中，那么会增加自旋时间
  • 对于某个锁，如果自旋很少成功获得锁，那么可能会减少自旋时间，甚至省略掉自旋过程

轻量级锁

目的：减少重量级锁对线程阻塞带来的线程开销。

在线程执行同步块之前，JVM会先在当前线程的栈帧中创建一个名为锁记录(Lock Record)的空间，用于存储锁对象目前的 Mark Word 的拷贝（JVM会将对象头中的 Mark Word 拷贝到锁记录中，官方称为 Displaced Mark Ward）这个时候线程堆栈与对象头的状态如图：

如上图，如果当前对象没有被锁定，那么锁标志位为 01 状态，JVM 在执行当前线程时，首先会在当前线程栈帧中创建锁记录 Lock Record 的空间用于存储锁对象目前的 Mark Word 的拷贝。

然后，虚拟机使用 CAS 操作将标记字段 Mark Word 拷贝到锁记录中，并且将 Mark Word 更新为指向 Lock Record 的指针。如果更新成功了，那么这个线程就拥用了该对象的锁，并且对象 Mark Word 的锁标志位更新为（Mark Word 中最后的 2bit）00，即表示此对象处于轻量级锁定状态，如图：

如果上述修改过程失败，JVM 就回去检查当前 Mark Word 中是否存在指向当前线程的栈帧的指针，如果有则直接调用，否则 CAS 自旋一定次数后膨胀为重量级锁，同时锁标志位改为 10。

偏向锁

多数情况下，多线程竞争的概率较低，而且总是对同一个线程反复加锁释放锁，由此，带来了很多不必要的开销和上下文的切换，由此引入偏向锁。

原理：当一个线程访问同步块并且获取锁时，会在对象头和栈帧中的锁记录里存储锁偏向的线程 ID，以后该线程进入和退出同步块时就不需要进行 CAS 操作来加锁解锁了。只需要简单地测试一下对象头里 Mark Word 是否存储着指向当前线程 ID 的偏向锁，如果成功，则表示已经获取到了锁。

偏向锁的撤销：出现等待竞争时才会释放锁。出现竞争时，暂停所有拥有偏向锁线程，检查持有偏向锁的线程是否存活：

  • 如果线程不处于活跃状态：偏向锁 -> 无锁
  • 如果线程或者：遍历栈帧中的锁记录，而栈帧中的锁记录和对象头要么偏向于其他线程，要么恢复到无锁状态或着膨胀为轻量级锁

锁的优缺点对比

synchronized vs Lock

synchronized 的不足：

  • 效率低：只有代码执行完毕后才会释放锁，不能设定中断、超时
  • 不灵活：每个锁仅有一个单一条件（某个对象），读写锁更灵活
  • 无法知道是否成功获得锁：相对于 Lock，加锁解锁操作不可见

Lock 的四个方法：

  • lock()：加锁
  • unlock()：解锁
  • tryLock()：尝试获取锁，返回 boolean 值
  • tryLock(long, TimeUtil)：尝试获取锁，可以设置超时

使用 synchronized 注意事项：

  • 锁对象头不能为空，因为锁信息都在对象头里
  • 作用域不宜过大，影响执行速度
  • 避免死锁
  • 尽量使用 JUC 里的类，如果非得二选一，synchronized 比 Lock 更优先一些，因为代码量少，避免出错
